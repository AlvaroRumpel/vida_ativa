# Coding Conventions

**Analysis Date:** 2026-03-19

## Naming Patterns

**Files:**
- Dart files use `snake_case`: `firebase_options.dart`, `widget_test.dart`
- Main entry point: `main.dart`
- Test files: `[name]_test.dart` or `[name]_widget_test.dart`

**Classes:**
- PascalCase for class names: `MyApp`, `DefaultFirebaseOptions`, `MyStatelessWidget`
- Classes follow Widget naming: `extends StatelessWidget`, `extends StatefulWidget`

**Functions:**
- camelCase for function names and methods: `main()`, `build()`, `initState()`, `dispose()`
- `async` functions clearly marked: `void main() async`, `Future<void> initializeApp()`

**Variables:**
- camelCase for local and instance variables: `context`, `tester`, `seedColor`
- Descriptive names preferred: `debugShowCheckedModeBanner` (not abbreviated)

**Constants:**
- PascalCase for class constants (Flutter convention): `Icons.add`, `Colors.green`, `DefaultFirebaseOptions.currentPlatform`
- camelCase for local constants: handled implicitly in const constructors

## Code Style

**Formatting:**
- Dart formatter (automatic via `dart format` or IDE integration)
- Default indentation: 2 spaces
- Single expressions fit on one line when possible
- Longer expressions broken across multiple lines

**Linting:**
- Tool: `flutter_lints` package (v6.0.0+)
- Config file: `analysis_options.yaml`
- Rules: `package:flutter_lints/flutter.yaml` included base ruleset
- Custom rules can override via `linter: rules:` section in `analysis_options.yaml`
- Run analysis: `flutter analyze`

**Key Active Rules:**
- `// ignore: name_of_lint` - disable single-line violations
- `// ignore_for_file: name_of_lint` - disable entire file
- Default Flutter lints enforced (good practices for Flutter apps)

## Import Organization

**Order:**
1. Dart SDK imports: `import 'dart:async';`, `import 'dart:io';`
2. Flutter framework imports: `import 'package:flutter/material.dart';`, `import 'package:flutter/foundation.dart';`
3. Package dependencies: `import 'package:firebase_core/firebase_core.dart';`, `import 'package:flutter_test/flutter_test.dart';`
4. Relative imports: `import 'firebase_options.dart';`, `import 'package:vida_ativa/main.dart';`

**Import conventions:**
- Use `package:` style imports for project files: `import 'package:vida_ativa/main.dart';` (not relative `../`)
- Selectively import using `show`: `import 'package:firebase_core/firebase_core.dart' show FirebaseOptions;`
- Use `as` for namespace aliasing when needed

## Error Handling

**Patterns:**
- Use `throw` with descriptive error messages
- Pattern: `throw UnsupportedError('Description of unsupported platform...');`
- Example from `firebase_options.dart`: Platform-specific error handling in switch statement
- Firebase initialization wrapped in `async/await` to handle initialization errors
- Widget error handling typically through `try/catch` or `FutureBuilder` widgets

**Validation:**
- Constructor parameters with `const` enforce immutability
- Use `required` keyword for mandatory parameters: `const MyApp({super.key});`
- Firebase options use platform detection (`kIsWeb`, `defaultTargetPlatform`) for validation

## Logging

**Framework:** `debugPrint()` for debug output (respects debug mode)

**Patterns:**
- Conditional debug output: `debugPrint('message');` instead of `print()`
- Use `avoid_print` lint rule to prevent debug prints in production
- Firebase initialization logging handled automatically by Firebase Core

## Comments

**When to Comment:**
- Auto-generated files marked with: `// File generated by FlutterFire CLI.`
- Suppress lints in generated files: `// ignore_for_file: type=lint`
- Class-level documentation above class definitions
- Complex logic documented with single-line comments

**JSDoc/Documentation:**
- Not extensively used in this project
- Generated files document purpose: `/// Default [FirebaseOptions] for use with your Firebase apps.`
- Example usage provided in comments for key classes

## Function Design

**Size:**
- Widget `build()` methods kept focused - delegate complex UI to sub-widgets
- `main()` function is minimal - defers initialization to methods
- Firebase initialization in separate function call

**Parameters:**
- Use named parameters in constructors: `DefaultFirebaseOptions({@required this.web})`
- Use `super.key` in Flutter widgets for proper key handling
- Constructor parameters marked as `const` when applicable

**Return Values:**
- Void functions for setup/initialization that don't return values: `void main()`, `void build()`
- Async functions return `Future`: `Future<void>`, `Future<Widget>`
- Widget `build()` returns `Widget` type
- Switch statements with complete case coverage (all paths must return or throw)

## Module Design

**Exports:**
- No barrel files (index.dart) currently used
- Each module imported directly: `import 'firebase_options.dart';`
- Firebase configuration exported from generated file

**Barrel Files:**
- Not used in current structure
- When needed, export re-exports would follow: `export 'src/models/user.dart';`

**Class Structure:**
- Single responsibility: `DefaultFirebaseOptions` only handles Firebase config
- `MyApp` StatelessWidget only builds the Material app root
- Test file focuses on single feature test

## Widget Conventions

**Stateless vs Stateful:**
- `MyApp` is `StatelessWidget` (no mutable state)
- Use `const` constructors: `const MyApp({super.key});`
- Immutable widget design encouraged

**Build Method:**
- Returns single root widget (typically `Scaffold`, `MaterialApp`, or custom widget)
- Accepts `BuildContext context` parameter
- Keep logic minimal - extract to separate methods or sub-widgets
