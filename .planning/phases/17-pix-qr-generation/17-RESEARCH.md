### Locked Decisions

**QR Display UX**
- After confirming booking: Navigator.push to dedicated **PixPaymentScreen** (not a sheet)
- BookingConfirmationSheet closes, PixPaymentScreen opens immediately

**Invoke da Cloud Function**
- Flutter calls `createPixPayment` via **httpsCallable** (FirebaseFunctions.instance.httpsCallable)
- Payload: `{ bookingId: String }`
- CF validates: `booking.userId == context.auth.uid` — rejects if mismatch
- CF saves PaymentRecord in `/bookings/{id}/payment/{txId}` AND returns QR data to Flutter
- If CF fails (MP unavailable): booking remains `pending_payment`; PixPaymentScreen shows error + "Tentar novamente" button that relaunches the callable

**Cloud Function — createPixPayment (Node.js)**
- Type: Callable Function (onCall v2)
- Credentials: **Firebase Secret Manager** via `defineSecret('MP_ACCESS_TOKEN')`
- Phase 17 uses **sandbox credentials** only
- Flow: validate auth → read booking (verify pending_payment + userId) → call MP Pix API → calculate expiresAt = now + 30min → save PaymentRecord → update booking with expiresAt + paymentId → return `{ qrCode, qrCodeBase64, expiresAt }`

**BookingModel new fields**
- `status` new values: `'pending_payment'` and `'expired'` (in addition to pending, confirmed, cancelled)
- `expiresAt: DateTime?` — Firestore Timestamp; when QR/slot expires
- `paymentId: String?` — MP transaction ID (set after createPixPayment)
- New getters: `isPendingPayment`, `isExpired`

**PaymentRecord schema — /bookings/{id}/payment/{txId}**
- `qrCode: String` — copia-e-cola string
- `qrCodeBase64: String` — base64 QR image (returned directly by MP)
- `expiresAt: Timestamp`
- `status: String` — 'pending' | 'paid' | 'expired'
- Document ID = txId from Mercado Pago (idempotency for Phase 18 webhook)

**QR Rendering in Flutter**
- Use `qrCodeBase64` with `Image.memory(base64Decode(qrCodeBase64))` — no extra package needed
- MP returns base64 directly from Pix API

**pending_payment in ScheduleScreen**
- Slots with `pending_payment` appear as **blocked** (same as confirmed)
- Other clients cannot book while slot is pending_payment
- ScheduleScreen must treat `pending_payment` as occupied in availability filter

**BookingCard in MyBookings**
- Status `pending_payment`: badge **"Aguardando pagamento"** + amber/orange color
- Status `expired`: badge **"Expirada"** + grey color (same as cancelled visually)
- Tap on `pending_payment` card: opens **PixPaymentScreen** reading PaymentRecord from subcollection
- PixPaymentScreen reused for initial flow AND via card tap

**Copia-e-cola UX**

## Summary

Phase 17 integrates Mercado Pago Pix QR code generation into the existing Flutter + Firebase booking flow. The project already has a solid callable function pattern (`notifyAdminNewBooking` as reference), a Firestore transaction-based booking creation (`bookSlot()`), and reactive stream state management via `BookingCubit`. The new `createPixPayment` callable function slots cleanly into this architecture.

The key architectural change is the `bookSlot()` flow: instead of writing `pending` or `confirmed` at booking time, it now always writes `pending_payment`, then Flutter immediately navigates to `PixPaymentScreen` which calls `createPixPayment`. The ScheduleCubit's whereIn filter (`['pending', 'confirmed']`) must be extended to include `pending_payment` so those slots remain blocked.

The Mercado Pago SDK v2.12.0 (`mercadopago` npm package) is the current version. It uses a `MercadoPagoConfig` + `Payment` class pattern. The Pix response returns `point_of_interaction.transaction_data.qr_code` (copia-e-cola string) and `qr_code_base64` (base64 image). Setting `date_of_expiration` to now + 30 min aligns with the project's 30-minute payment window requirement. Secrets are managed via `defineSecret('MP_ACCESS_TOKEN')` with `secrets: [mpAccessToken]` in the function config.

**Primary recommendation:** Keep the bookSlot() Firestore transaction as-is but change `initialStatus` to always be `'pending_payment'` (bypassing the existing confirmationMode logic for Pix flow). Call `createPixPayment` CF immediately after successful `bookSlot()` in the confirmation sheet, then Navigator.push to PixPaymentScreen with the bookingId.

### Anti-Patterns to Avoid

- **Double payment creation:** Always use `bookingId` as `idempotencyKey` in MP API call. If CF is called twice for same booking, MP returns existing payment instead of creating duplicate.
- **Missing `secrets: [mpAccessToken]`:** If the array is omitted, `mpAccessToken.value()` returns empty string inside the function — silent failure.
- **Auto-cancel for pending_payment:** ScheduleCubit has auto-cancel logic for `isPending` bookings past their time. This must NOT apply to `pending_payment` — Phase 18 CF handles expiration. Add guard: `if (booking.isPending)` (existing check) already excludes `pending_payment` because `isPending` checks `status == 'pending'`.
- **Navigating with context after async gap:** `BookingConfirmationSheet._handleConfirm()` pops the sheet then pushes `PixPaymentScreen`. Must check `mounted` before each navigation call.

## Common Pitfalls

### Pitfall 1: MP Sandbox Payer Email Required
**What goes wrong:** MP Pix API in sandbox requires a valid `payer.email` field. Omitting it returns a 400 error.
**Why it happens:** MP validates payer identity even in sandbox mode.
**How to avoid:** Pass the booking user's email to the CF (either from Firestore `/users/{userId}` doc or from `request.auth.token.email` if available).
**Warning signs:** CF returns `"bad_request"` or `"invalid payer"` from MP API.

### Pitfall 2: date_of_expiration Format
**What goes wrong:** MP requires ISO 8601 format with timezone offset, not just UTC Z suffix in some environments.
**Why it happens:** MP API is strict about date format.
**How to avoid:** Use `expiresAt.toISOString()` in Node.js (produces correct format). Verify in sandbox before assuming it works.
**Warning signs:** MP returns 400 with `"invalid date_of_expiration"`.

### Pitfall 3: ScheduleCubit auto-cancel guard
**What goes wrong:** The existing `_recompute()` in ScheduleCubit auto-cancels `pending` bookings past their slot time. If `pending_payment` bookings are accidentally treated as `pending`, they get auto-cancelled.
**Why it happens:** `isPending` getter currently checks `status == 'pending'`. As long as the new getter is `isPendingPayment` and the auto-cancel block stays `if (booking.isPending)`, this is safe. But if someone changes the condition to `!booking.isConfirmed`, it would break.
**How to avoid:** Keep the auto-cancel guard as `booking.isPending` (exact status match). Add a comment explaining `pending_payment` is excluded intentionally.

### Pitfall 4: Missing `mercadopago` in functions/package.json
**What goes wrong:** CF deploys but throws `Cannot find module 'mercadopago'` at runtime.
**Why it happens:** `npm install` must be run in `functions/` before deploy.
**How to avoid:** Add to `functions/package.json` dependencies, run `cd functions && npm install`, commit `package-lock.json`.

### Pitfall 5: PixPaymentScreen opened twice (initial + card tap)
**What goes wrong:** When tapping `pending_payment` card in MyBookings, the screen needs to load PaymentRecord from Firestore subcollection — it cannot use the in-memory CF result.
**How it happens:** Initial flow receives QR data from CF response. Card tap flow must fetch from `/bookings/{id}/payment/{txId}` subcollection.
**How to avoid:** PixPaymentScreen accepts either (a) pre-loaded QR data (from initial flow) or (b) just `bookingId` and fetches PaymentRecord on init. Use a factory constructor or optional parameters.

### Pitfall 6: notifyAdminNewBooking fires on pending_payment
**What goes wrong:** Existing CF `notifyAdminNewBooking` triggers on new bookings with `status != 'cancelled'`. A `pending_payment` booking will trigger admin push notification.
**Why it happens:** `isActive` check is `after.status !== 'cancelled'` — `pending_payment` passes this check.
**How to avoid:** Either acceptable behavior (admin sees pending_payment notifications) or add `pending_payment` exclusion. Decision: acceptable — admin should know about new bookings regardless. Document this as expected behavior.

## Code Examples

### Cloud Function — createPixPayment (complete structure)

```javascript
// functions/index.js — add after existing notifyAdminNewBooking export
const { onCall } = require('firebase-functions/v2/https');
const { defineSecret } = require('firebase-functions/params');
const { MercadoPagoConfig, Payment } = require('mercadopago');

const mpAccessToken = defineSecret('MP_ACCESS_TOKEN');

exports.createPixPayment = onCall(
  { secrets: [mpAccessToken] },
  async (request) => {
    if (!request.auth) {
      throw new Error('unauthenticated');
    }

    const { bookingId } = request.data;
    const callerId = request.auth.uid;

    const db = admin.firestore();

    // 1. Read and validate booking
    const bookingRef = db.collection('bookings').doc(bookingId);
    const bookingSnap = await bookingRef.get();
    if (!bookingSnap.exists) throw new Error('booking_not_found');

    const booking = bookingSnap.data();
    if (booking.userId !== callerId) throw new Error('permission_denied');
    if (booking.status !== 'pending_payment') throw new Error('invalid_status');

    // 2. Get payer email (from auth token or users collection)
    const payerEmail = request.auth.token.email || `user_${callerId}@sandbox.test`;

    // 3. Call Mercado Pago
    const client = new MercadoPagoConfig({
      accessToken: mpAccessToken.value(),
      options: { timeout: 10000 },
    });
    const paymentApi = new Payment(client);

    const now = new Date();
    const expiresAt = new Date(now.getTime() + 30 * 60 * 1000);

    const result = await paymentApi.create({
      body: {
        transaction_amount: booking.price,
        payment_method_id: 'pix',
        date_of_expiration: expiresAt.toISOString(),
        payer: { email: payerEmail },
        description: `Reserva ${bookingId}`,
        external_reference: bookingId,
      },
      requestOptions: { idempotencyKey: bookingId },
    });

    const txId = String(result.id);
    const qrCode = result.point_of_interaction.transaction_data.qr_code;
    const qrCodeBase64 = result.point_of_interaction.transaction_data.qr_code_base64;

    // 4. Save PaymentRecord subcollection
    await db
      .collection('bookings').doc(bookingId)
      .collection('payment').doc(txId)
      .set({
        qrCode,
        qrCodeBase64,
        expiresAt: admin.firestore.Timestamp.fromDate(expiresAt),
        status: 'pending',
        createdAt: admin.firestore.FieldValue.serverTimestamp(),
      });

    // 5. Update booking with paymentId and expiresAt
    await bookingRef.update({
      paymentId: txId,
      expiresAt: admin.firestore.Timestamp.fromDate(expiresAt),
    });

    // 6. Return QR data to Flutter
    return {
      qrCode,
      qrCodeBase64,
      expiresAt: expiresAt.toISOString(),
    };
  }
);
```

## Open Questions

1. **Payer email in sandbox**
   - What we know: MP Pix API requires `payer.email`. Firebase Auth email may be null (anonymous/phone-only users).
   - What's unclear: Does this project use email-based auth or phone/anonymous?

   - What we know: Existing CF fires when `status != 'cancelled'`. `pending_payment` will trigger it.
   - What's unclear: Whether admin wants notifications for unconfirmed Pix bookings.
   - Recommendation: Leave as-is for Phase 17. Admin seeing "new booking awaiting payment" is useful signal. Phase 18 may refine.

   - What we know: Project uses go_router (GoRouter import visible in my_bookings_screen.dart).
   - What's unclear: Whether PixPaymentScreen needs a named route or Navigator.push is sufficient.
